# UI 视觉精修任务书

本文档基于当前功能基本完整的版本，继续规划下一轮 UI 精修。目标不是增加大功能，而是把现有界面从“控件都能用”提升到“长期使用舒服、信息层级清晰、状态反馈明确、视觉上更像一个成熟工作台”。

## 当前版本观察

当前 UI 已经完成了多项关键改进：

- 已有全局状态栏和全局工具栏。
- 已有“环境配置”Tab。
- 已内置 MapleMono-CN 字体，中文和英文使用同一套等宽风格字体。
- 已把运行、单步、停止流程、启动/关闭托管 Chrome 放到全局位置。
- 进度文字已经从裸 `newTruncatingLabel` 改为专用 `progressLabel`。
- 多个列表、表格、日志和产物路径已经支持右键菜单。

但当前界面仍有明显精修空间：

1. 顶部状态栏、全局工具栏和 Tab 区域视觉层级偏弱，看起来像默认控件堆叠。
2. 全局工具栏内容较多，在窗口变窄时容易拥挤，按钮组和选择器缺少明确边界。
3. 多个页面的标题、操作按钮、筛选区和主体内容排列方式不统一。
4. 步骤表没有表头，列含义依赖用户猜测，尤其是状态、启用、等待、失败处理等列。
5. 流程库、历史记录、环境列表仍是单行文本，信息密度低但可读性不够强。
6. 步骤属性面板表单很长，字段没有按“定位 / 输入 / 等待 / 失败处理 / 备注”分组。
7. 运行详情页的日志区、摘要区和产物区视觉权重不均，日志行颜色存在但层级不够。
8. 设置页过于表单化，危险项、路径项、下载项没有明显分区。
9. 主题颜色当前整体偏浅灰 + 蓝色，缺少更细的中性色层级、边框层级和状态色规范。
10. 自定义主题当前没有充分按 `variant` 区分浅色/深色；如果系统进入深色模式，可能出现对比度问题。

## 总体方向

这个应用是流程编排和运行工具，应继续保持工作台式、安静、清晰的风格：

- 不做营销页、渐变大背景、装饰插画或花哨动画。
- 不使用大圆角卡片堆叠，不把页面做成卡片套卡片。
- 优先提升信息层级、对齐、间距、状态表达和常用操作可见性。
- 保持 Fyne 原生控件体验，但通过主题、容器、分组、表头、状态徽标和空状态增强专业度。
- 所有视觉优化必须保证小窗口可用，不能让按钮、Tab、表格文字互相挤压变形。

## P0-1. 建立统一的视觉 Token 和 UI Helper

### 问题

当前颜色和布局主要散落在各文件中，例如：

- `internal/ui/theme.go`
- `internal/ui/status_bar.go`
- `internal/ui/run_panel.go`
- `internal/ui/labels.go`

状态色、文本色、背景色、间距、标题样式、分组样式没有统一入口，后续继续改 UI 会越来越难保持一致。

### 具体方案

新增或扩展一个 UI helper 文件，例如：

```text
internal/ui/style.go
```

建议统一提供：

```go
func uiColorSurface() color.Color
func uiColorPanel() color.Color
func uiColorBorder() color.Color
func uiColorMutedText() color.Color
func uiColorSuccess() color.Color
func uiColorWarning() color.Color
func uiColorDanger() color.Color
func uiColorInfo() color.Color

func newPageTitle(title string, subtitle string) fyne.CanvasObject
func newSectionHeader(title string, actions ...fyne.CanvasObject) fyne.CanvasObject
func newToolbarGroup(title string, controls ...fyne.CanvasObject) fyne.CanvasObject
func newStatusBadge(text string, kind statusKind) fyne.CanvasObject
func newMutedLabel(text string) *widget.Label
```

注意：

- Helper 不是为了制造复杂抽象，而是为了让页面标题、分组标题、状态徽标、弱化文本、工具栏分组一致。
- Fyne 没有 CSS，需要用少量 `canvas.Rectangle` + `container.NewStack` 或 `container.NewBorder` 做轻量背景和边界。
- 不要到处手写 RGB 值，状态色都走统一函数。

### 验收标准

- 新增页面或修改页面时，不需要重复手写标题样式、状态色和分组布局。
- 状态栏、运行日志、历史状态、步骤状态使用同一套状态色语义。
- 全局 UI 的边框、背景、弱文本颜色一致。

## P0-2. 精修主题，补齐浅色/深色模式和对比度

### 问题

当前 `internal/ui/theme.go` 的 `Color()` 对很多颜色直接返回浅色值，基本没有根据 `variant` 区分浅色和深色。若 Fyne 跟随系统深色主题，可能出现：

- 背景仍是浅色，但部分默认控件按深色渲染。
- 文本、按钮、输入框对比度不稳定。
- 选择、高亮、禁用态观感不一致。

### 具体方案

1. 明确产品策略：
   - 如果暂时只做浅色模式，应确保所有主题颜色都稳定为浅色体系，并测试系统深色模式下不会混色。
   - 如果支持深色模式，则 `Color(name, variant)` 必须为 `theme.VariantDark` 单独返回完整颜色。

2. 建议浅色颜色层级：

```text
窗口背景：#F6F7F9
工具栏/页面表面：#FFFFFF
次级表面：#F0F3F7
边框：#D8DEE8
主文本：#20242A
弱文本：#69717D
主色：#2563EB
成功：#16A34A
警告：#D97706
危险：#DC2626
信息：#2563EB
```

3. 深色模式如果实现，建议：

```text
窗口背景：#111318
工具栏/页面表面：#181B21
次级表面：#20242C
边框：#303642
主文本：#E6E8EC
弱文本：#A4ABB6
主色：#60A5FA
成功：#4ADE80
警告：#FBBF24
危险：#F87171
```

4. 重点覆盖这些 Fyne theme color：

```go
theme.ColorNameBackground
theme.ColorNameForeground
theme.ColorNamePrimary
theme.ColorNameButton
theme.ColorNameDisabled
theme.ColorNameHover
theme.ColorNameSelection
theme.ColorNameInputBackground
theme.ColorNameSeparator
theme.ColorNamePlaceHolder
theme.ColorNameError
```

5. 字号保持当前基线即可，不要继续变小：

```go
Text: 14
Caption: 12
Heading: 16
Padding: 8
InlineIcon: 18
```

若要提升精致度，优先通过间距和层级调整，不要盲目缩小字体。

### 验收标准

- 浅色模式下页面不再像默认灰底控件堆叠。
- 深色系统环境下没有低对比度文字、白底黑字混乱或输入框不可读。
- 状态色在浅色/深色下都可辨认。

## P0-3. 重构顶部状态栏和全局工具栏的视觉层级

### 问题

当前顶部区域信息很多：

- 当前流程
- 保存状态
- Chrome 状态
- 运行状态
- 流程选择
- 保存
- 启动/关闭 Chrome
- 运行/单步/停止
- 环境选择
- 进度条和进度文字

这些信息全部排在顶部，功能上正确，但视觉上容易拥挤，且各组边界不够清晰。

### 具体方案

1. 把顶部区域做成两个明确的横向带：

```text
状态带：产品名 | 当前流程 | 保存 | Chrome | 运行
操作带：流程选择 | 浏览器组 | 执行组 | 环境选择
进度带：当前步骤文本 + 进度条
```

2. `statusBar` 建议：
   - 产品名左侧保留。
   - 状态项改成紧凑徽标：字段名弱化，值突出。
   - 状态点尺寸固定，例如 8x8。
   - 每个状态项固定宽度，避免长流程名挤压其他项。

3. `globalToolbar` 建议：
   - `newToolbarGroup("流程", ...)` 给流程、浏览器、执行、环境四组建立一致边界。
   - 工具栏组之间用固定间距，不用过多 `widget.Separator`。
   - 主要按钮只有“运行”使用高强调。
   - “停止流程”“关闭托管”保持危险色，但不要两个危险按钮紧挨造成误点，可中间隔开或放到不同组。

4. 窄窗口策略：
   - 流程选择固定宽度 220-260。
   - 环境选择固定宽度 160-200。
   - 进度文字固定宽度，长步骤名只截断步骤名，不截断 `第 N/M 步`。
   - 当窗口小于 1280 宽时，可把进度带放到下一行，不要压缩按钮文字到不可读。

5. 文案建议：

```text
启动 Chrome
关闭托管
运行
单步
停止流程
```

“单步执行”在顶部可以缩短为“单步”，运行后切换为“下一步”。

### 验收标准

- 顶部区域看起来是有层级的控制台，而不是一长串按钮。
- 1280x720 下按钮和选择框不重叠、不显示异常省略。
- 用户能一眼分清状态、操作和进度。

## P0-4. 给步骤表增加表头、状态徽标和更清晰的列宽策略

### 问题

当前 `internal/ui/step_table.go` 使用 `widget.Table`，没有表头。列含义只能通过经验猜：

```text
序号 / 状态 / 启用 / 名称 / 类型 / 目标 / 输入 / 等待 / 失败处理
```

这会降低可读性，尤其是新用户很难理解 `✓`、空白列和毫秒列。

### 具体方案

1. 优先改为 `widget.NewTableWithHeaders`。
2. 表头建议：

```text
# | 状态 | 启用 | 步骤名称 | 类型 | 目标 | 输入/期望 | 等待 | 失败处理
```

3. 状态列不要只显示字符，建议统一状态徽标：

```text
运行中：蓝色圆点 + 运行
成功：绿色 ✓
失败：红色 ×
跳过：灰色 -
```

Fyne 表格单元格如果不方便放复杂组件，也至少把状态字符和颜色封装成专用 cell renderer。

4. 列宽建议：

```text
#：44
状态：72
启用：56
步骤名称：180
类型：120
目标：260
输入/期望：220
等待：80
失败处理：110
```

窗口较小时允许横向滚动，不要把所有列硬压到可视宽度内。

5. 对禁用步骤：
   - 整行弱化。
   - 启用列显示“停用”或灰色标识，不只留空。

6. 选中步骤后，`selectedBar` 的按钮可以改为小工具栏：

```text
已选中第 3 步：复制 | 上移 | 下移 | 删除
```

删除按钮放最后，危险色。

### 验收标准

- 用户不看文档也能理解每一列含义。
- 长 XPath、输入内容不会污染其他列。
- 禁用、成功、失败、运行中的步骤能快速区分。

## P1-1. 重做流程库列表项的信息结构

### 问题

当前流程库每行只显示：

```text
流程名称 [tag1, tag2]
```

搜索和标签筛选可用，但列表项信息层级较弱，选中状态和流程元信息不明显。

### 具体方案

1. 每个流程列表项建议显示两行：

```text
流程名称
描述摘要 / 标签 / 步骤数量 / 最近运行状态
```

2. 标签显示为小文本或轻量徽标，避免 `[tag1, tag2]` 挤在名称后面。
3. 若没有描述，显示弱文本：

```text
无描述
```

4. 搜索区优化：
   - 搜索框和标签筛选并排或紧凑排列。
   - 顶部按钮只保留“新建”，其他操作放“更多”和右键菜单。

5. 选中项需要有更明显的高亮背景，不要只依赖 Fyne 默认弱高亮。

### 验收标准

- 流程库能快速扫描名称、标签和描述。
- 长流程名不会导致列表行变形。
- 选中项在浅色和深色下都清晰。

## P1-2. 重构步骤属性面板，按任务语义分组

### 问题

当前 `internal/ui/step_property.go` 中表单字段很多，虽然会按步骤类型动态显示，但所有字段仍在一个长表单里，用户需要自己理解字段关系。

### 具体方案

1. 将表单拆成分组标题：

```text
基础信息
定位目标
输入与模板
等待与超时
失败处理
备注
```

2. 不同步骤类型只显示相关分组：
   - 打开网址：基础信息 / 目标 URL / 等待与超时 / 失败处理 / 备注
   - 点击元素：基础信息 / 定位目标 / 等待与超时 / 失败处理 / 备注
   - 输入文本：基础信息 / 定位目标 / 输入与模板 / 等待与超时 / 失败处理 / 备注
   - 断言文本：基础信息 / 定位目标 / 期望文本 / 等待与超时 / 失败处理 / 备注

3. 模板预览区：
   - 当前 `模板预览：xxx，yyy` 容易挤压。
   - 建议改为固定高度的预览区域，最多显示 3 行。
   - “插入模板 / 预览 / 校验”三项做成同一行工具条，宽度固定。

4. `应用到当前步骤`：
   - 保持底部固定可见。
   - 如果字段有错误，按钮附近显示错误摘要。
   - 应用成功后给短暂状态提示，例如“已应用”。

5. 数字输入项：
   - 等待、超时字段建议使用统一宽度，例如 120。
   - 标签中保留单位 `ms`，输入框内只放数字。

### 验收标准

- 用户编辑步骤时能按分组理解字段。
- 长模板和预览不会挤压右侧属性面板。
- 应用按钮始终容易找到。

## P1-3. 优化环境配置页的表格与操作区

### 问题

环境配置页功能已完整，但环境列表和变量表仍偏基础。变量表中的“操作”列当前为空，容易让用户困惑。

### 具体方案

1. 如果操作主要通过右键和顶部按钮完成，删除空的“操作”列。
2. 变量表列建议：

```text
KEY | VALUE | 类型 | 说明
```

其中“类型”显示：

```text
普通 / 敏感
```

3. 敏感变量：
   - VALUE 显示 `******`。
   - 类型列显示危险或警告语义的徽标，例如“敏感”。

4. 环境列表项两行化：

```text
环境名 [当前]
变量数量 / 说明摘要
```

5. 导入/导出按钮：
   - 放在环境配置页右上角的“配置文件”组。
   - 不要和“新增变量”混在一起，避免用户误以为导入的是单个变量。

6. 空状态：
   - 没有变量时显示“当前环境暂无变量”，并提供“新增变量”“导入配置”。

### 验收标准

- 环境页不再出现空操作列。
- 敏感变量能一眼识别。
- 导入/导出语义清楚，是环境配置级别操作。

## P1-4. 优化运行详情页的信息布局

### 问题

运行详情页现在左侧是日志，右侧是摘要、当前步骤和产物。功能齐全，但摘要区偏弱，日志区缺少结构化层级。

### 具体方案

1. 顶部进度区保留：

```text
第 N/M 步 · 步骤名
进度条
```

2. 右侧摘要改成固定宽度信息面板：

```text
运行摘要
成功 12
失败 0
跳过 1
耗时 8.3s

当前步骤
输入用户名

产物
截图路径
HTML 路径
```

3. 日志行优化：
   - 时间弱化。
   - 状态关键词使用统一状态色。
   - 错误日志可加左侧细色条或前缀。
   - 日志字号保持 13，不要更小。

4. 日志工具按钮：
   - `清空日志` 是破坏性操作，放在右侧或更多菜单最后。
   - `复制日志`、`打开产物目录` 保持可见。

5. 空日志状态：

```text
暂无运行日志
```

运行开始后自动消失。

### 验收标准

- 用户能快速看到运行是否成功、当前卡在哪一步、产物在哪里。
- 错误日志比普通日志更醒目。
- 日志区大量内容时仍能顺畅滚动。

## P1-5. 优化历史页为可扫描记录列表

### 问题

当前历史记录每行是单行文本：

```text
06-12 15:04:05 | success | 成功:10 失败:0
```

信息可用但不够直观。

### 具体方案

1. 历史项改为两行：

```text
成功 · 2026-06-12 15:04:05 · 8.3s
环境：测试环境 · 成功 10 / 失败 0 / 跳过 0
```

2. 状态显示为徽标：
   - 成功：绿色
   - 失败：红色
   - 运行中或异常中断：蓝/黄，若有该状态

3. 筛选区更紧凑：

```text
环境 [全部环境]   状态 [全部状态]
```

4. 无历史时显示空状态：

```text
暂无执行历史
```

并提示用户可以运行当前流程生成历史。

### 验收标准

- 历史记录不再像日志文本，能够快速扫描成功/失败和耗时。
- 右键菜单功能保持不变。

## P1-6. 优化设置页的信息分区

### 问题

设置页目前只有一个表单，路径、下载源、自定义 URL、缓存策略混在一起。

### 具体方案

1. 按语义拆成分区：

```text
Chrome 来源
安装与数据目录
下载校验
缓存策略
```

2. 对关键约束做短提示，但不要堆大段说明：

```text
已有本地 Chrome 时不会重新下载。
每次重放都会使用隔离 profile。
```

3. 路径输入框：
   - 安装目录、用户数据目录宽度更大。
   - 可增加“选择目录”按钮。

4. 自定义 ZIP 区域只在选择“自定义 ZIP”时显示，并带明确边界。

5. 保存成功的状态提示不要只在底部一行文字里显示，建议靠近保存按钮或使用状态徽标。

### 验收标准

- 设置页能一眼区分下载源、目录和缓存策略。
- 自定义下载配置出现/隐藏时布局不跳动。

## P2-1. 全局空状态与错误状态规范

### 问题

多个页面在没有数据时会直接显示空白区域，用户难以判断是未加载、没有数据还是出错。

### 具体方案

统一空状态组件：

```go
func newEmptyState(title, message string, action fyne.CanvasObject) fyne.CanvasObject
```

覆盖场景：

- 没有任何流程。
- 当前没有选择流程。
- 当前流程没有步骤。
- 当前环境没有变量。
- 没有运行历史。
- 运行详情页暂无日志。
- 产物目录为空。

错误状态组件：

```go
func newInlineError(message string) fyne.CanvasObject
```

用于表单校验、加载失败、导入失败等场景。

### 验收标准

- 用户不会面对空白区域不知道下一步做什么。
- 空状态文案简短，提供一个明确操作。

## P2-2. 统一按钮层级和文案

### 问题

当前按钮已经使用图标，但不同页面按钮层级仍不完全一致。

### 具体方案

1. 按钮重要性规范：

```text
HighImportance：新建、运行、保存配置、应用步骤
MediumImportance：保存、导入、导出、复制
DangerImportance：删除、停止流程、关闭托管 Chrome
普通按钮：筛选、预览、校验、更多
```

2. 同一页面按钮顺序：

```text
主要操作 -> 次要操作 -> 更多 -> 危险操作
```

3. 文案保持短：

```text
新建
保存
导入
导出
运行
单步
停止流程
关闭托管
更多
```

4. 危险操作：
   - 删除流程、删除环境、删除变量必须确认。
   - 停止流程和关闭托管 Chrome 不必每次确认，但视觉上要和运行按钮拉开距离。

### 验收标准

- 同类型操作在不同页面看起来一致。
- 主要操作明确，危险操作不会误触。

## P2-3. 验收截图与窗口尺寸矩阵

### 问题

UI 精修如果只在一个窗口尺寸下看，会遗漏小窗口挤压、字体省略、弹框变形等问题。

### 具体方案

后续 Agent 完成 UI 修改后，至少检查以下尺寸：

```text
1280x720
1366x768
1440x900
1920x1080
```

重点检查：

- 顶部状态栏和工具栏是否挤压。
- Tab 文本是否完整。
- 步骤表是否可读，是否需要横向滚动。
- 步骤属性表单是否能滚动到底部。
- 环境配置页左右分栏比例是否合理。
- 弹框输入框是否足够宽。
- 进度文字是否再次变成 `...`。
- 中文字体是否仍为 MapleMono-CN 风格。

如果本地 Linux 缺少 Fyne/GLFW 图形依赖，至少完成代码审查；具备 Windows 环境时必须实际启动二进制检查。

### 验收标准

- 上述窗口尺寸下没有文字重叠、按钮被挤没、输入框过窄、表格列错乱。
- 视觉修改不破坏现有流程运行、环境管理、导入导出和右键菜单功能。

## 建议实施顺序

1. 先做 `style.go` 和主题颜色补齐，统一视觉基础。
2. 再精修顶部状态栏和全局工具栏，因为它们影响所有页面。
3. 然后做步骤表表头和步骤属性分组，这是最高频编辑流程。
4. 接着优化环境配置、运行详情、历史和设置页。
5. 最后统一空状态、按钮层级，并做多尺寸验收。

## 非目标

本轮不要做以下事情：

- 不要新增大功能，除非是 UI 精修所需的轻量 helper。
- 不要引入 WebView 或前端框架重写界面。
- 不要引入运行期网络依赖。
- 不要提交 `data/`、`logs/`、`chrome/` 或构建产物。
- 不要为了美观牺牲当前已完成的全局操作、环境配置、右键菜单、字体和进度修复。
